# 问答与检索架构

## 处理链路

```text
用户问题
  -> QueryAnalysisService（规范化、实体识别、必要时拆分）
  -> RetrievalService（BM25 + BGE 向量检索）
  -> 等权 RRF 融合、正文去重、依赖上下文补齐
  -> ContextBuilderService（正文与完整元数据）
  -> LLM 生成带 evidence_id 的结构化回答
  -> CitationValidatorService（引用与效力约束）
  -> API / SSE / Web UI
```

所有入口最终调用 `ComplianceService.answer()` 或 `answerStream()`，不存在另一套绕过检索的回答逻辑。

## 问题分析

`QueryAnalysisService` 先做 NFKC 规范化、简称替换和错别名归一，再识别产品、主体、监管机构、时间、效力问题与监管主题。复杂问题按标点和并列关系最多拆成 5 个子问题；简单问题保持单条。

同义词扩展采用有限词表。例如“雪球”只扩展到场外期权、自动赎回、复杂金融产品、挂钩标的和投资者适当性，不递归扩展。

## 混合检索

- BM25：对法规标题、章节、条号和正文等权建索引，不设置字段权重。
- 向量：`Xenova/bge-base-zh-v1.5` ONNX q8，768 维，本地推理。
- 融合：BM25 与向量各取 Top 30，采用相同参数的 RRF：`1 / (60 + rank)`。
- 精确匹配：用户明确输入完整法规名或文号时作为结果锚点，不改变一般查询的字段权重。
- 去重：按 Chunk ID、规范化正文哈希和重叠来源去重；遇到“前款、上述规定”等依赖表达时补充相邻上下文。

向量模型不提交仓库。首次运行 `pnpm download:retrieval-model` 下载固定版本；模型缺失时 API 会记录降级日志并返回 BM25 结果，不伪装成已执行混合检索。

## 上下文

每条证据包括 `evidence_id`、法规名称、发文主体、文号、发布日期、施行日期、效力状态、章节、条款、原文、官网 URL、本地来源、文档 ID、Chunk ID、检索方式和分数。模型不能看到缺少归属信息的孤立正文。

## 回答约束

回答模型只允许返回带 `evidence_id` 的 JSON。强结论包括“可以、不得、必须、应当、禁止、仅限、需要备案、仍然有效、已废止”等，必须关联本次证据。

回答采用四类标签：

- `【明确规定】`
- `【基于法规的推导】`
- `【需结合具体业务判断】`
- `【证据不足】`

证据不足时固定输出：“根据当前知识库检索结果，暂时无法形成确定结论。”

## 引用校验

`CitationValidatorService` 不信任模型生成的法规元数据。它以 `evidence_id` 查找本次命中，并用知识库中的法规名、条号、URL 和效力状态覆盖模型字段。以下情况会将回答降级为“需人工合规复核”：

- 引用不在本次检索结果中；
- 强结论没有有效证据；
- 效力问题所引文件状态为空或 `unknown`；
- 已废止或失效文件被当作现行依据；
- 引用全部失效或无法核验。

URL 为空时保持为空，校验器不会生成链接。

## 异常处理

- 索引未完成加载：API 返回 `INDEX_NOT_READY`。
- 向量模型缺失：明确记录 BM25 降级。
- LLM 未配置：只返回检索线索和证据不足提示。
- LLM 返回非法 JSON：请求失败并保留错误，不吞掉异常。
- URL 回写冲突：写入 `outputs/url_update_conflicts.csv`，不自动覆盖。
