# 受控法规问答智能体架构

## 一、核心思路

系统已从“规则扩展后固定检索一次，再让模型回答”改为一个有上限、可审计的工具型智能体：

```text
用户问题
  -> 规则基础规范化（只作兜底）
  -> 规划模型生成检索计划
  -> hybrid_regulation_search
  -> 规划模型判断证据充分性
  -> 必要时补充检索一次
  -> 回答模型生成结构化回答与逐字引文
  -> 程序化真实性校验
  -> 独立审查模型
  -> 必要时修订或补充检索
  -> 最终回答
```

模型只能提出计划、选择证据和生成结构化内容。检索工具参数校验、执行、次数上限、错误处理、元数据回填和最终降级由 Harness 掌控。

## 二、有限状态机

`AgentWorkflowService` 显式执行以下状态：

1. `ANALYZE_QUERY`：NFKC 规范化、主体/产品/主题基础识别。
2. `PLAN_RETRIEVAL`：调用规划模型，生成子问题、正式术语、检索表达和必需证据。
3. `RETRIEVE`：Harness 校验工具输入并执行混合检索。
4. `ASSESS_EVIDENCE`：判断是否有直接规定、例外、主体和时间条件。
5. `RETRIEVE_AGAIN`：第一轮有可检索缺口时补充检索。
6. `DRAFT_ANSWER`：回答模型仅读取最终证据并选择 `evidenceId`。
7. `VERIFY_DETERMINISTICALLY`：程序校验引用、原文、数字、效力与适用范围。
8. `REVIEW_ANSWER`：独立审查模型输出 `PASS` / `REPAIR` / `RETRIEVE`。
9. `REPAIR_OR_RETRIEVE`：依据错误类型修订或补充证据。
10. `FINALIZE`：返回结果或“不能确认，需要人工合规复核”。

硬上限是：最多两轮检索、一次回答修订、两次审查（初审＋修订后复审）和 24 次状态转换。任何分支都不能无限循环。

## 三、三类模型调用

### 1. 规划与证据判断

Prompt：`packages/prompts/agent/retrieval-planner.md`。

第一次调用只生成检索计划，不回答最终法律问题。检索后再用同一角色判断直接规定、一般要求、例外、主体、交易对手和时间条件是否齐全。

原有 `QueryAnalysisService` 保留，但仅用于：

- 基础规范化；
- 安全同义词补充；
- 规划模型无效 JSON 时降级；
- 防止模型漏掉“敲入、敲出、本公司股票、未来生效”等关键安全方向。

### 2. 证据约束回答

Prompt：`packages/prompts/agent/evidence-answer.md`。

回答模型只能读取最终证据集。它输出“是、否或不能确认”、适用范围、限制、缺失信息、`evidenceId` 和 `quoteExact`。它不输出法规标题、文号、条款或 URL。

每条证据还会附带由系统从原 Chunk 中截取的短连续原文候选。这些候选只帮助模型逐字复制，不改变检索排名，也不是人工标注的相关 Chunk。

### 3. 独立证据审查

Prompt：`packages/prompts/agent/answer-reviewer.md`。

审查模型不生成新答案，不能引入新法规。它只判断：

- 证据是否真正支持结论；
- 是否把一般条件当成明确许可；
- 是否扩大主体、交易对手或产品范围；
- 是否遗漏例外和生效时间；
- 是否跨监管制度套用定义或比例规则；
- 问题需要文字修订还是新证据。

## 四、混合检索工具

`HybridRegulationSearchTool` 将现有检索封装为只读语义工具 `hybrid_regulation_search`。

输入必须通过 Zod 校验，包括检索表达、子问题、主体、产品、时间、必需证据和 `topK`。工具没有写入文件、调用外部业务系统或修改索引的权限。

工具内部继续使用：

- BM25 Top 30；
- 本地 `Xenova/bge-base-zh-v1.5` q8 向量 Top 30；
- `1 / (60 + rank)` 等权 RRF；
- Chunk 去重；
- overlap、引导语和前置条款补齐。

多个子问题分别检索时，各批结果的归一化分数不能直接横向比较。最终证据合并按子问题轮转保留，避免前一个宽泛子问题挤掉后一个直接规则子问题的关键条文。

返回字段包含 `evidenceId`、Chunk/文档 ID、法规元数据、原文、BM25/向量/RRF 排名、上下文标记和子问题。普通用户界面不展示这些内部 ID 与排名。

## 五、程序校验与独立审查的分工

### 程序必须确定的事

`CitationValidatorService` 检查：

- `evidenceId` 是否属于本次证据；
- Chunk 和文档是否真实；
- `quoteExact` 是否为对应 Chunk 中的连续原文；
- 字词、比例、金额、日期和期限是否被修改；
- 标题、文号、条款和 URL 是否来自正式元数据；
- 废止或尚未施行文件是否被误当现行规则；
- 是否把不同主体制度的条款拼成一条适用链。

校验可以忽略空白、换行和全半角 ASCII 差异定位原始片段，但最终展示的引文仍由系统从原 Chunk 连续切取。改一个字、25%、1000 万元、日期或期限都会失败。

### 模型审查的事

独立审查不重复判断 ID 和字符串是否存在，而是检查“这条原文是否真的支持这个结论”。程序校验不通过时不调用审查模型。

## 六、引用和用户界面

模型只选择 `evidenceId` 和原文；系统回填法规名称、发文机关、文号、条款、效力状态和官网 URL。

普通页面展示：

1. 是 / 否 / 不能确认；
2. 准确结论与结论层级；
3. 主体、产品、交易对手、时间和条件；
4. 限制、例外和尚缺信息；
5. 法规名称、条款、效力状态和官网链接；
6. 与模型解释分开的逐字法规原文。

Chunk ID、`evidenceId`、BM25/向量/RRF 排名、状态 trace 和审查意见只在 `debug: true` 的 API 调用和本地评测产物中返回，不在普通前端展示。

## 七、错误与降级

- 规划 JSON 无效：使用规则分析降级计划。
- 工具参数无效：返回结构化工具错误，不崩溃状态机。
- 向量模型不可用：记录 BM25 降级。
- 无 `LLM_API_KEY`：可执行规则分析和检索，但必须返回“不能确认”及明确降级状态。
- 回答或审查 JSON 无效：安全归一可识别的字段；无法归一则降级。
- DeepSeek 返回429、503、529等瞬时过载：保持原模型，最多额外重试两次并采用指数退避；不会切换为低配模型或无限重试。
- 达到两轮检索、一次修订或两次审查上限：不再循环，返回人工复核。

## 八、生产与评测是同一条链

NestJS 非流式 API、SSE API、Web 前端和 `scripts/evaluate_retrieval.mjs` 全部调用 `ComplianceService`，再进入同一个 `AgentWorkflowService`。评测集不预设 Chunk ID，只检查最终结论、必需法规、关键限定、逐字引文、官网 URL、程序校验和独立审查。

评测结果写入被 Git 忽略的 `data/index/eval/results.json` 和 `results.md`，不将长模型上下文提交到仓库。
