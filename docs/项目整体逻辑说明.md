# 中国场外衍生品法规知识库：项目整体逻辑说明

## 一、这个项目要解决什么问题

这个项目的目标，不是让大模型凭记忆回答法规问题，而是先从本地法规库中找到真正相关的条文，再让大模型严格依据这些条文整理答案，并把法规名称、条款和官网链接一并展示出来。

整条链路可以概括为：

```text
法规原件
→ 结构化正文
→ 按法规结构切成 Chunk
→ 建立关键词索引和向量索引
→ 用户提问
→ DeepSeek Pro 规划需要查什么
→ 调用混合检索工具查找相关 Chunk
→ 判断证据是否充分，必要时补充检索
→ DeepSeek Pro 基于最终证据生成回答
→ 系统校验引用、逐字原文、效力和适用范围
→ 独立模型审查证据是否真正支持结论
→ 返回“直接回答 + 解释 + 法规依据 + 官网链接”
```

因此，这个系统的核心不是“模型知道多少法规”，而是三件事：

1. 法规原件和结构化正文是否准确；
2. 用户提问后能否找回真正关键的条文；
3. 模型是否只依据找到的条文回答，没有虚构法规或扩大结论。

## 二、法规库里有什么

### 1. 当前规模

当前正式知识库包括：

- 108份法规原件；
- 108份结构化正文；
- 1,221个正式 Chunk；
- 108份法规均配置了官网链接；
- 1,221个 Chunk ID 全部唯一，当前没有漏审或失去来源的 Chunk。

原件、结构化文本和 Chunk 的关系是一一可追溯的：每个 Chunk 都能定位到所属法规和对应的结构化原文块。

当前数据可以通过两个不需要后端服务的静态页面查看：

- [监管文件总目录](../data/processed/chunk_review_viewer/regulation_catalog.html)：查看全部法规的名称、文号、发文主体、日期、效力状态、本地原件和官网来源；
- [Chunk 切分查看器](../data/processed/chunk_review_viewer/chunk_review.html)：查看每份法规的 Chunk 正文、条款范围和切分结果。

这两个页面可以：

- 搜索法规名称、文号和发文机关；
- 查看每份法规的基本信息和官网链接；
- 查看每份法规被切成了多少个 Chunk；
- 查看每个 Chunk 的正文和条款范围；
- 在法规目录和 Chunk 正文之间相互跳转；
- 通过链接定位或复制具体 Chunk。

## 三、法规原件如何变成结构化正文

原始法规包括 PDF、DOC、DOCX 等格式。不同格式不能直接用同一种方式读取，因此项目先进行格式解析，再统一整理成结构化 JSON。

结构化阶段主要完成以下工作：

1. 提取正文，同时保留原始文件名称和来源路径；
2. 识别法规标题、文号、发文机关、发布日期、施行日期和效力状态；
3. 识别章、节、条、款、项、附件和表格；
4. 删除真正的目录、重复页眉页脚和孤立页码；
5. 保留 Word 表格、文本框和 smartTag 中的有效正文；
6. 检查 PDF 是否存在扫描页、乱码、私有区字符和公式丢失；
7. 给每个结构化原文块分配 `source_block_id`，供后面的 Chunk 追溯。

这里有一个重要原则：原始法规文件只读，不在原件上直接修改。所有格式解析、正文清洗和层级整理都发生在结构化层。因此，结构化 JSON 不只是一段从原件中复制的文字，而是带有来源、层级和元数据的可追溯法规正文。

## 四、Chunk 是怎么切的

### 1. 为什么需要 Chunk

一份法规可能有几十页。如果每次提问都把整份法规交给模型：

- 上下文太长；
- 相关条文容易被淹没；
- 检索粒度太粗；
- 模型很难精确引用具体条款。

因此，需要把法规切成若干个相对独立、又保留必要上下文的 Chunk。

### 2. 先识别法规结构，再决定边界

项目不是简单地“每隔固定字数切一刀”，而是先识别法规结构：

```text
文件或分部
  └─ 章
      └─ 节
          └─ 条
              └─ 款
                  └─ 项、子项
附件、表格和说明单独识别
```

切分时优先选择以下边界：

- 新文件或新分部开始；
- 新章、新节开始；
- 完整条款结束；
- 完整款、项或列表结束；
- 附件开始；
- 主题已经明显变化。

### 3. 长度只是约束，不是第一原则

当前参数大致是：

- 目标长度约600个正文字符；
- 一般上限约1,200个正文字符；
- 少于200字符的普通短块，优先和相邻内容合并。

但是，系统不会为了凑够600字把两个无关章节硬合并，也不会为了严格控制1,200字而从一句话中间切断。

例如：

- 一条法规只有150字，但内容是完整的禁止性条款，可以独立保留；
- 一条法规有1,500字，优先在完整句子或完整分项处拆开；
- 一个表格过长时按完整表格行拆分，不从单元格中间截断；
- 单个无法继续拆分的完整语义单元超过上限时，会保留并明确标记原因，而不是破坏原意。

### 4. 引导语不能和列表分离

下列内容原则上应放在同一个 Chunk，或者通过上下文补充保持关系：

```text
应当符合下列条件：
（一）……
（二）……
（三）……
```

不能只把“应当符合下列条件”留在前一个 Chunk，而让后一个 Chunk 从“（一）”孤立开始。否则模型虽然看到了列表项，却不知道这些列表项是在说明条件、禁止事项还是例外。

### 5. overlap 不是固定重复一段文字

本项目没有采用机械的固定滑窗重叠。只有在后一个 Chunk 明显依赖前文时，才补充必要上下文，例如：

- “依照前款”；
- “除前项规定外”；
- “上述情形”；
- 一个长列表被拆成两块；
- 后文是前一条定义或条件的延续。

补充时最多参考前面两个完整语义单元，并控制补充内容不要占据后一个 Chunk 的大部分。这样既能避免语义断裂，也能减少无意义重复。

### 6. 每个 Chunk 保留什么信息

每个 Chunk 不只有正文，还会保留：

- `chunk_id` 和文档内顺序；
- `document_id`；
- 法规名称、文号和发文机关；
- 发布日期、施行日期和效力状态；
- 章、节、条款范围和附件名称；
- 官网链接；
- 对应的 `source_block_id`；
- 是否为重叠 Chunk、重叠来源和是否超长。

因此，模型检索到一个 Chunk 时，不会只看到一段失去出处的文字。

## 五、关键词检索是怎么工作的

关键词检索使用 BM25。通俗地说，它会判断：

- 用户问题中的词是否在某个 Chunk 中出现；
- 这些词在该 Chunk 中出现得是否集中；
- 这些词是不是在整个法规库中比较有区分度。

例如用户问“私募产品投资雪球比例”，关键词检索会重点关注：

- 私募证券投资基金；
- 雪球；
- 场外期权或收益凭证；
- 投资比例、上限；
- 基金净资产。

为了适应中文法规文本，系统会进行中文双字切分，并额外识别“场外衍生品”“收益凭证”“终止净额”“专业投资者”等领域词。法规名称、文号、章、条和正文都会进入关键词索引，但不会仅因为某个发文机关级别更高就人为加分。

BM25 的优点是：对法规名称、文号、固定术语、数字和明确措辞很敏感。缺点是：如果用户使用口语表达，而法规使用另一种正式说法，关键词可能对不上。

## 六、向量检索是怎么工作的

向量检索解决的是“表达不同，但意思接近”的问题。

例如用户问：

> 券商收益凭证可以做雪球吗？

法规中不一定出现完全相同的句子，但可能出现：

- 证券公司发行浮动收益凭证；
- 挂钩标的；
- 内嵌衍生品；
- 收益情景分析；
- 敲入和敲出结构。

向量模型会把用户问题和每个 Chunk 转成一组代表语义的数字，再比较语义距离。

当前使用的向量模型是：

`Xenova/bge-base-zh-v1.5`，768维，q8量化版本。

它在本地运行，不需要把全部法规发送给外部模型。向量文件与1,221个 Chunk 严格一一对应。Chunk 内容发生变化后，向量必须重新生成，不能继续使用旧向量。

向量检索的优点是能理解近义表达和口语问题；缺点是有时会找到“主题很像、但法律上不能直接支持结论”的内容。因此它不能单独决定最终法律答案。

## 七、为什么要做混合检索工具

关键词检索和向量检索各有长处：

| 检索方式 | 更擅长 |
|---|---|
| BM25关键词检索 | 法规名称、文号、条号、数字、固定法律术语 |
| 向量检索 | 口语问题、近义表达、概念关联和语义相似内容 |

每次工具调用内部都会同时执行两种检索：BM25 和 BGE 各取前30条，再用 RRF 融合、去重并补齐必要的前置条款。RRF 可以理解为“排名投票”：同一个 Chunk 在两类检索中都靠前，融合后通常也更靠前；它不直接相加两套口径不同的原始分数。

与旧的固定流程不同，现在不是把用户原问题扩词后机械检索一次。DeepSeek Pro 会先生成检索计划：

1. 判断用户问的是许可、禁止、比例、定义还是效力；
2. 识别主体、产品、交易对手、时间和歧义；
3. 把宽泛问题拆成必要的子问题；
4. 把“券商”“雪球”等口语转换为有边界的法规正式表达；
5. 说明回答每个子问题需要直接规定、例外、主体限定还是生效时间；
6. 调用只读工具 `hybrid_regulation_search`；
7. 阅读第一轮证据并判断缺了什么；
8. 只有存在明确、可检索的证据缺口时，才补充检索一次。

检索计划不是人工标注的 Chunk 清单。模型只能决定“查什么”，不能指定一个并未检索到的引用。原有规则词表仍保留，但只用于规范化、安全同义词补充和规划 JSON 失败时兜底，不再主导整条问答链。

工具参数必须通过结构校验，`topK` 只能在4至20之间。工具只能读取现有索引，不能修改法规、Chunk 或其他业务数据。无论模型怎样规划，整个问题最多检索两轮，避免无休止搜索。

## 八、受控问答智能体如何工作

### 1. 使用什么模型

当前大模型配置为 `deepseek-v4-pro`，通过 OpenAI-compatible 接口调用；本地向量模型仍是 `Xenova/bge-base-zh-v1.5`。

两者分工不同：

- BGE 只把问题和 Chunk 转成向量，帮助工具找材料；
- DeepSeek Pro 分别承担检索规划、证据判断、回答和独立审查。

这里所谓“智能体”不是让模型任意操作系统，而是让它在有限状态机中作决定。程序掌握工具权限、调用次数、错误处理、引用回填和最终降级。

### 2. 第一阶段：规划与证据判断

规划 Prompt 要求模型先确定主体、产品、交易对手和时间，再输出严格 JSON。它不能在规划阶段回答法律问题。

检索完成后，模型再次判断：现有材料有没有直接规定、一般要求、例外、主体限定和生效时间。第一轮证据足够就停止检索；存在明确缺口时才生成第二轮查询。证据不充分且已没有检索次数时，必须留下“缺什么”的说明，不能用模型记忆补齐。

### 3. 第二阶段：基于证据回答

回答模型只看到最终检索证据和证据充分性摘要。Prompt 的核心规则是：

1. 先回答“是、否或不能确认”，再解释准确范围；
2. 每个重要结论必须绑定本次证据中的真实 `evidenceId`；
3. `quoteExact` 必须从单个 Chunk 连续逐字复制，不能改字、拼接或用省略号；
4. 不得把一般合规条件说成具体产品的明确许可；
5. 不得把局部禁止扩大成所有主体一律禁止；
6. 不得把未来规则当成现行规则；
7. 不得把一类产品或主体的定义、比例和例外套到另一套制度；
8. 证据不足时回答“不能确认”，并说明缺失信息。

模型只输出引用 ID、逐字原文和该证据支持什么，不自行生成法规标题、文号、条款号和 URL。系统根据真实元数据统一回填这些字段，避免伪造官网链接或记错条号。

### 4. 第三阶段：程序校验

回答初稿先经过确定性校验。程序检查：

- 引用是否真的来自本次工具返回；
- 逐字引文是否是对应 Chunk 的连续原文；
- 数字、比例、金额、日期、期限和主体名称是否被改动；
- 标题、文号、条款、效力状态和官网 URL 是否来自正式元数据；
- 已废止或尚未施行的规则是否被错误当成现行依据；
- 是否把私募证券基金规则跨制度套给集合资产管理计划等其他主体。

空格、换行及全半角 ASCII 的纯格式差异可以被定位回原文，但最终展示的仍是 Chunk 中真实的连续片段。改一个字、把25%改成其他比例、改日期或金额都会校验失败。

### 5. 第四阶段：独立模型审查

程序校验通过后，再调用独立审查 Prompt。审查模型不写新答案，也不能引入新法规，只判断：证据是否真的支持候选结论，范围是否被扩大，是否漏掉例外和生效时间，以及是否需要新证据。

审查结果只有三类：

- `PASS`：证据和结论一致；
- `REPAIR`：已有证据足够，但文字、范围或引用需要修订；
- `RETRIEVE`：缺少直接规则、例外或效力证据，需要补检。

整个问题最多两轮检索、一次回答修订、两次审查和24次状态转换。达到任一上限仍不能形成可靠答案时，系统返回“不能确认，需要人工合规复核”，不会无限循环。

### 6. 最终如何返回

普通页面只展示业务人员真正需要的内容：

```text
直接回答：是 / 否 / 不能确认
准确结论与结论层级
适用主体、产品、交易对手、时间和条件
限制、例外和尚缺信息
法规名称、条款、效力状态、官网链接
与模型解释分开的逐字法规原文
```

Chunk ID、BM25/向量/RRF 排名、状态轨迹和审查意见只在 `debug: true` 的接口响应和本地评测中出现，不污染普通用户界面。非流式接口、SSE、前端和评测脚本都进入同一个 `ComplianceService` 和 `AgentWorkflowService`，不存在两套问答逻辑。

## 九、三个真实问题的回答示例

下面列的是首版三道真实问题的目标回答口径，也是评测脚本检查的业务边界。评测不预置 Chunk ID；真实运行时仍由智能体规划检索并从实际证据中选择引用。

### 示例一：上市公司可以做挂钩自己本身标的的场外衍生品吗？

**直接回答：否，不能笼统认定可以。**

现行规则下，证券公司和期货风险管理公司这两条主要交易对手路径均有针对上市公司自身股票标的的禁止规定。对于其他持牌交易对手，仍需结合其现行适用规则判断，不能把已确认路径的禁止扩大成“所有交易对手一律禁止”。

同时，《衍生品交易监督管理办法（试行）》已经公布，但要到2026年11月16日才施行。该办法施行后，将更广泛地禁止上市公司达成以其发行股票为标的物的衍生品交易，法定例外除外。它可以用于说明未来规则，但目前不能当作已经生效的现行依据。

主要依据：

- [《证券公司场外期权业务管理办法》](https://www.sac.net.cn/zlgl/zlgz/202512/t20251231_69845.html)：规定证券公司场外期权路径下的相关禁止要求；
- [《期货风险管理公司衍生品交易业务管理规则》](https://www.cfachina.org/governmentrules/selfdisciplinerules/businessspecificationclass/202502/t20250224_79001.html)：第四十八条规定特定交易路径下的禁止要求；
- [《衍生品交易监督管理办法（试行）》](https://www.csrc.gov.cn/csrc/c101953/c7632837/content.shtml)：第十六条规定未来更广泛的禁止，并需注明尚未施行及生效日期。

### 示例二：券商收益凭证可以做雪球吗？

**直接回答：不能确认。**

现有《证券公司收益凭证发行管理办法》和配套披露指引规定了浮动收益凭证的业务资格、挂钩标的、亏损约束、专业投资者、信息披露和内嵌衍生品风控要求，但没有直接写明“收益凭证可以采用雪球结构”，也没有对雪球类收益凭证设置一条独立的普遍授权。

因此，系统不能只凭一般性规则回答“明确可以”。具体产品仍需要结合发行证券公司的业务资格、敲入敲出安排、挂钩标的、损失结构和监管口径判断。

主要依据：

- [《证券公司收益凭证发行管理办法》](https://www.sac.net.cn/sjb/zlgl/zlgz/202512/t20251231_69939.html)：规定浮动收益凭证的发行、销售和合规风控要求；
- [《证券公司收益凭证信息披露指引》](https://www.sac.net.cn/tzg/202512/t20251231_78190.html)：要求发行说明书披露结构信息、收益情景和对冲干扰事件等。

这里的“不能确认”不是说法规明确禁止，而是说现有直接证据不足以作出“普遍明确允许”的结论。

### 示例三：私募产品投资雪球比例的范围有明确规定吗？

**直接回答：是。**

《私募证券投资基金运作指引》第十七条规定，私募证券投资基金参与证券公司等机构发行的带敲入和敲出结构的场外期权或者收益凭证，例如雪球结构衍生品，其合约名义本金原则上不得超过基金净资产的25%。

例外是：全部投资者均为符合规定的专业投资者、穿透后单个投资者投资金额不低于1,000万元，并且基金为封闭式私募证券投资基金。满足全部条件时，可以适用条文规定的例外。

主要依据：

- [《私募证券投资基金运作指引》](https://fg.amac.org.cn/governmentrules_3854/zcgz_zlgz/zlgz_smjj/zlgz_smjj_cpl/202404/t20240430_25509.html)：第十七条明确规定25%上限及例外条件。

需要注意，“私募产品”是一个宽泛说法。该条直接适用的是私募证券投资基金，不能未经核实就自动套用到所有私募股权基金或所有私募资产管理计划。

## 十、如何理解这个项目的可靠性

这个系统不是因为“大模型很聪明”才可靠，而是通过分层约束降低错误：

```text
原件负责提供真实内容
结构化层负责保持正文和层级
Chunk 层负责提供合适的检索单位
BM25 负责找准关键词和数字
向量模型负责理解相近表达
RRF 负责融合两类检索
规划模型负责拆解问题和发现证据缺口
检索工具负责只读、可控地取得材料
回答模型负责依据最终证据组织回答
程序校验负责阻止伪造引用、改字改数和效力错误
独立审查模型负责检查证据是否真正支持结论
端到端评测负责确认真实用户问题走通了整条链路
```

当前自动测试和复核结果包括：

- 108份正式法规；
- 1,221个 Chunk；
- 1,221个本地向量；
- 1,221/1,221 Chunk 完成复核；
- 本地测试包括7项检索测试、31项受控智能体测试和47项文档处理测试；
- 3个真实问题已使用 `deepseek-v4-pro` 走正式生产链路重新评测，结果为3/3；
- 三题的程序引用校验和独立审查均通过，实际模型调用分别为4次、6次和4次。

但这仍然是法规检索和证据整理系统，不替代律师法律意见或金融机构内部合规审批。对于主体、产品结构、交易对手、生效日期或监管口径不明确的问题，系统应当明确展示证据边界，而不是为了给出一个简单答案而过度推断。
