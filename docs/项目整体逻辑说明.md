# 中国场外衍生品法规知识库：项目整体逻辑说明

## 一、这个项目要解决什么问题

这个项目的目标，不是让大模型凭记忆回答法规问题，而是先从本地法规库中找到真正相关的条文，再让大模型严格依据这些条文整理答案，并把法规名称、条款和官网链接一并展示出来。

整条链路可以概括为：

```text
法规原件
→ 结构化正文
→ 按法规结构切成 Chunk
→ 建立关键词索引和向量索引
→ 用户提问
→ 混合检索相关 Chunk
→ DeepSeek Pro 基于检索证据生成回答
→ 系统校验引用、效力状态和结论
→ 返回“直接回答 + 解释 + 法规依据 + 官网链接”
```

因此，这个系统的核心不是“模型知道多少法规”，而是三件事：

1. 法规原件和结构化正文是否准确；
2. 用户提问后能否找回真正关键的条文；
3. 模型是否只依据找到的条文回答，没有虚构法规或扩大结论。

### 1. 当前规模

- 108份法规原件；

- 1,221个 Chunk ID 全部唯一，当前没有漏审或失去来源的 Chunk。

原件、结构化文本和 Chunk 的关系是一一可追溯的：每个 Chunk 都能定位到所属法规和对应的结构化原文块。

## 三、法规原件如何变成结构化正文

原始法规包括 PDF、DOC、DOCX 等格式。不同格式不能直接用同一种方式读取，因此项目先进行格式解析，再统一整理成结构化 JSON。

结构化阶段主要完成以下工作：

1. 提取正文，同时保留原始文件名称和来源路径；
2. 识别法规标题、文号、发文机关、发布日期、施行日期和效力状态；
3. 识别章、节、条、款、项、附件和表格；
4. 删除真正的目录、重复页眉页脚和孤立页码；
5. 保留 Word 表格、文本框和 smartTag 中的有效正文；
6. 检查 PDF 是否存在扫描页、乱码、私有区字符和公式丢失；
7. 给每个结构化原文块分配 `source_block_id`，供后面的 Chunk 追溯。

这里有一个重要原则：原始法规文件只读，不在原件上直接修改。所有清洗和修复都发生在结构化层。

在复核过程中还专门修复了三类转换问题：

- 旧 DOC 中目录后的签约前言不能和目录一起被误删；
- 利率定义文件中的贴现公式保留为 `λ = 1 / [1 + (r × N / D)]`；
- 信用衍生品估值文件中的分式、指数、求和和积分，按原 PDF 的二维排版转成运算顺序明确的一维表达。

因此，结构化 JSON 不只是“从 PDF 复制出来的一段文字”，而是一份带有来源、层级和元数据的可追溯法规正文。

## 四、Chunk 是怎么切的

### 1. 为什么需要 Chunk

一份法规可能有几十页。如果每次提问都把整份法规交给模型：

- 上下文太长；
- 相关条文容易被淹没；
- 检索粒度太粗；
- 模型很难精确引用具体条款。

因此，需要把法规切成若干个相对独立、又保留必要上下文的 Chunk。

### 2. 先识别法规结构，再决定边界

项目不是简单地“每隔固定字数切一刀”，而是先识别法规结构：

```text
文件或分部
  └─ 章
      └─ 节
          └─ 条
              └─ 款
                  └─ 项、子项
附件、表格和说明单独识别
```

切分时优先选择以下边界：

- 新文件或新分部开始；
- 新章、新节开始；
- 完整条款结束；
- 完整款、项或列表结束；
- 附件开始；
- 主题已经明显变化。

### 3. 长度只是约束，不是第一原则

当前参数大致是：

- 目标长度约600个正文字符；
- 一般上限约1,200个正文字符；
- 少于200字符的普通短块，优先和相邻内容合并。

但是，系统不会为了凑够600字把两个无关章节硬合并，也不会为了严格控制1,200字而从一句话中间切断。

例如：

- 一条法规只有150字，但内容是完整的禁止性条款，可以独立保留；
- 一条法规有1,500字，优先在完整句子或完整分项处拆开；
- 一个表格过长时按完整表格行拆分，不从单元格中间截断；
- 单个无法继续拆分的完整语义单元超过上限时，会保留并明确标记原因，而不是破坏原意。

### 4. 引导语不能和列表分离

下列内容原则上应放在同一个 Chunk，或者通过上下文补充保持关系：

```text
应当符合下列条件：
（一）……
（二）……
（三）……
```

不能只把“应当符合下列条件”留在前一个 Chunk，而让后一个 Chunk 从“（一）”孤立开始。否则模型虽然看到了列表项，却不知道这些列表项是在说明条件、禁止事项还是例外。

### 5. overlap 不是固定重复一段文字

本项目没有采用机械的固定滑窗重叠。只有在后一个 Chunk 明显依赖前文时，才补充必要上下文，例如：

- “依照前款”；
- “除前项规定外”；
- “上述情形”；
- 一个长列表被拆成两块；
- 后文是前一条定义或条件的延续。

补充时最多参考前面两个完整语义单元，并控制补充内容不要占据后一个 Chunk 的大部分。这样既能避免语义断裂，也能减少无意义重复。

### 6. 每个 Chunk 保留什么信息

每个 Chunk 不只有正文，还会保留：

- `chunk_id` 和文档内顺序；
- `document_id`；
- 法规名称、文号和发文机关；
- 发布日期、施行日期和效力状态；
- 章、节、条款范围和附件名称；
- 官网链接；
- 对应的 `source_block_id`；
- 是否为重叠 Chunk、重叠来源和是否超长。

因此，模型检索到一个 Chunk 时，不会只看到一段失去出处的文字。

## 五、关键词检索是怎么工作的

关键词检索使用 BM25。通俗地说，它会判断：

- 用户问题中的词是否在某个 Chunk 中出现；
- 这些词在该 Chunk 中出现得是否集中；
- 这些词是不是在整个法规库中比较有区分度。

例如用户问“私募产品投资雪球比例”，关键词检索会重点关注：

- 私募证券投资基金；
- 雪球；
- 场外期权或收益凭证；
- 投资比例、上限；
- 基金净资产。

为了适应中文法规文本，系统会进行中文双字切分，并额外识别“场外衍生品”“收益凭证”“终止净额”“专业投资者”等领域词。法规名称、文号、章、条和正文都会进入关键词索引，但不会仅因为某个发文机关级别更高就人为加分。

BM25 的优点是：对法规名称、文号、固定术语、数字和明确措辞很敏感。缺点是：如果用户使用口语表达，而法规使用另一种正式说法，关键词可能对不上。

## 六、向量检索是怎么工作的

向量检索解决的是“表达不同，但意思接近”的问题。

例如用户问：

> 券商收益凭证可以做雪球吗？

法规中不一定出现完全相同的句子，但可能出现：

- 证券公司发行浮动收益凭证；
- 挂钩标的；
- 内嵌衍生品；
- 收益情景分析；
- 敲入和敲出结构。

向量模型会把用户问题和每个 Chunk 转成一组代表语义的数字，再比较语义距离。

当前使用的向量模型是：

`Xenova/bge-base-zh-v1.5`，768维，q8量化版本。

它在本地运行，不需要把全部法规发送给外部模型。向量文件与1,221个 Chunk 严格一一对应。Chunk 内容发生变化后，向量必须重新生成，不能继续使用旧向量。

向量检索的优点是能理解近义表达和口语问题；缺点是有时会找到“主题很像、但法律上不能直接支持结论”的内容。因此它不能单独决定最终法律答案。

## 七、为什么要做混合检索

关键词检索和向量检索各有长处：

| 检索方式 | 更擅长 |
|---|---|
| BM25关键词检索 | 法规名称、文号、条号、数字、固定法律术语 |
| 向量检索 | 口语问题、近义表达、概念关联和语义相似内容 |

系统会同时运行两种检索：

1. 问题先经过规范化，识别主体、产品和监管主题；
2. 根据有限词表做受控扩展，例如“券商”转换为“证券公司”，“雪球”关联到自动赎回、敲入敲出和场外期权；
3. BM25 对关键词问题取前30条；
4. 向量检索对语义问题取前30条；
5. 使用 RRF 融合两个排名；
6. 从融合结果前部组装最多14个 Chunk；
7. 如果某个 Chunk 依赖“前款”或 overlap 来源，再补入必要的前置上下文，但总量仍控制在14个以内。

RRF 可以理解为一种“排名投票”：一个 Chunk 如果同时在关键词检索和向量检索中排名靠前，融合后通常也会更靠前。它不直接比较两种检索完全不同的分数，因此比简单相加更稳定。

最终交给回答模型的是这批实际选出的 Chunk，不是某个人工预设的“标准 Chunk”。

## 八、大模型如何基于 Chunk 回答

### 1. 使用什么模型

回答模型使用 DeepSeek Pro，当前配置名称为：

`deepseek-v4-pro`

需要区分两个模型的职责：

- BGE 是本地向量模型，只负责“找材料”；
- DeepSeek Pro 是回答模型，负责“阅读材料、判断证据边界并组织答案”。

### 2. Prompt 的核心要求

Prompt 没有要求模型背诵法规，而是对回答过程施加以下约束：

1. 只能依据本次检索到的证据回答；
2. 每个重要结论必须绑定真实的 `evidence_id`；
3. 不得自行补充法规名称、文号、条款、效力状态和网址；
4. “可以、不得、必须、禁止”等强结论必须有直接证据；
5. 已废止文件不能作为现行依据；
6. 已公布但尚未施行的文件只能说明未来规则；
7. 不得把同一条中针对不同产品的条件交叉套用；
8. 在结论或限制条件中提到的法规必须正式引用；
9. 证据不足时要明确说“不能确认”，不能猜测；
10. 先给直接判断，再给解释和法规依据。

模型需要返回结构化 JSON，主要包括：

- `directAnswer`：是、否或不能确认；
- `conclusion`：对直接判断的解释；
- `regulatoryBasis`：模型选择的 `evidence_id` 和该证据支持什么；
- `restrictions`：限制条件；
- `missingInfo`：仍然缺少的信息；
- `manualReviewNote`：需要人工合规复核的事项。

### 3. 模型选择引用，系统回填真实信息

模型只负责从本次14个以内的证据中选择 `evidence_id`。系统随后根据该 ID 回填：

- 法规正式名称；
- 发文机关；
- 条款范围；
- 原文摘录；
- 效力状态；
- 官网链接。

这样可以避免模型自己“写出一个看起来像真的链接”或者把条款编号记错。

### 4. 回答如何校验

模型生成初稿后，系统会检查：

- 引用的 `evidence_id` 是否真的在本次检索结果里；
- 强结论是否有直接证据；
- 法规是否现行有效；
- 尚未施行的法规有没有被误说成已经生效；
- 上市公司自身股票问题有没有区分交易对手和生效时间；
- 收益凭证通用规定有没有被错误扩大成“明确允许所有雪球结构”；
- 同一个证据有没有被模型重复引用。

如果初稿没有通过校验，系统不会立刻把错误答案返回给用户，而是把：

- 原回答；
- 具体校验问题；
- 同一批检索证据

重新交给 DeepSeek Pro 修订一次。修订后再次校验。第二次仍不能通过时，系统才会降级为“不能确认，需要人工复核”。

这一步的作用不是让模型搜索新材料，而是让它重新检查已经拿到的材料，补齐漏引、收窄过强结论。

### 5. 最终如何返回

最终页面按照以下顺序展示：

```text
直接回答：是 / 否 / 不能确认

为什么

基于以下法规：
1. 法规名称 + 条款 + 官网链接 + 该条支持什么
2. ……

限制条件

需要补充的信息或人工复核提示
```

前端非流式接口、流式接口和端到端评测都调用同一个问答服务，不存在“评测时一套逻辑、用户真实提问时另一套逻辑”。

## 九、三个真实问题的回答示例

下面三题不是单纯的关键词排名测试，而是完整走过正式问题分析、混合检索、DeepSeek Pro 回答和引用校验的真实问答。当前结果为3/3通过。

### 示例一：上市公司可以做挂钩自己本身标的的场外衍生品吗？

**直接回答：否，不能笼统认定可以。**

现行规则下，期货风险管理公司不得与上市公司开展以该上市公司自身股票为标的的衍生品交易。对于其他持牌交易对手，仍需结合现行适用规则判断，不能把特定交易路径的禁止扩大成所有交易一律禁止。

同时，《衍生品交易监督管理办法（试行）》已经公布，但要到2026年11月16日才施行。该办法施行后，将更广泛地禁止上市公司达成以其发行股票为标的物的衍生品交易，法定例外除外。它可以用于说明未来规则，但目前不能当作已经生效的现行依据。

主要依据：

- [《期货风险管理公司衍生品交易业务管理规则》](https://www.cfachina.org/governmentrules/selfdisciplinerules/businessspecificationclass/202502/t20250224_79001.html)：第四十八条规定特定交易路径下的禁止要求；
- [《衍生品交易监督管理办法（试行）》](https://www.csrc.gov.cn/csrc/c101953/c7632837/content.shtml)：第十六条规定未来更广泛的禁止，并需注明尚未施行及生效日期。

### 示例二：券商收益凭证可以做雪球吗？

**直接回答：不能确认。**

现有《证券公司收益凭证发行管理办法》和配套披露指引规定了浮动收益凭证的业务资格、挂钩标的、亏损约束、专业投资者、信息披露和内嵌衍生品风控要求，但没有直接写明“收益凭证可以采用雪球结构”，也没有对雪球类收益凭证设置一条独立的普遍授权。

因此，系统不能只凭一般性规则回答“明确可以”。具体产品仍需要结合发行证券公司的业务资格、敲入敲出安排、挂钩标的、损失结构和监管口径判断。

主要依据：

- [《证券公司收益凭证发行管理办法》](https://www.sac.net.cn/sjb/zlgl/zlgz/202512/t20251231_69939.html)：规定浮动收益凭证的发行、销售和合规风控要求；
- [《证券公司收益凭证信息披露指引》](https://www.sac.net.cn/tzg/202512/t20251231_78190.html)：要求发行说明书披露结构信息、收益情景和对冲干扰事件等。

这里的“不能确认”不是说法规明确禁止，而是说现有直接证据不足以作出“普遍明确允许”的结论。

### 示例三：私募产品投资雪球比例的范围有明确规定吗？

**直接回答：是。**

《私募证券投资基金运作指引》第十七条规定，私募证券投资基金参与证券公司等机构发行的带敲入和敲出结构的场外期权或者收益凭证，例如雪球结构衍生品，其合约名义本金原则上不得超过基金净资产的25%。

例外是：全部投资者均为符合规定的专业投资者、穿透后单个投资者投资金额不低于1,000万元，并且基金为封闭式私募证券投资基金。满足全部条件时，可以适用条文规定的例外。

主要依据：

- [《私募证券投资基金运作指引》](https://fg.amac.org.cn/governmentrules_3854/zcgz_zlgz/zlgz_smjj/zlgz_smjj_cpl/202404/t20240430_25509.html)：第十七条明确规定25%上限及例外条件。

需要注意，“私募产品”是一个宽泛说法。该条直接适用的是私募证券投资基金，不能未经核实就自动套用到所有私募股权基金或所有私募资产管理计划。

## 十、如何理解这个项目的可靠性

这个系统不是因为“大模型很聪明”才可靠，而是通过分层约束降低错误：

```text
原件负责提供真实内容
结构化层负责保持正文和层级
Chunk 层负责提供合适的检索单位
BM25 负责找准关键词和数字
向量模型负责理解相近表达
RRF 负责融合两类检索
DeepSeek Pro 负责阅读证据和组织回答
引用校验负责阻止无依据或效力错误的结论
端到端评测负责确认真实用户问题走通了整条链路
```

当前自动测试和复核结果包括：

- 108份正式法规；
- 1,221个 Chunk；
- 1,221个本地向量；
- 1,221/1,221 Chunk 完成复核；
- 3个真实问题端到端评测全部通过。

但这仍然是法规检索和证据整理系统，不替代律师法律意见或金融机构内部合规审批。对于主体、产品结构、交易对手、生效日期或监管口径不明确的问题，系统应当明确展示证据边界，而不是为了给出一个简单答案而过度推断。
