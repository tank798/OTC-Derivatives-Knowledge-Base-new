# 对话式法规 Agent 逻辑

## 一、整体思路

问答层不再是一条写死的“问题分类→拆分子问题→固定多轮模型→独立审查”流水线。

现在是一套持续理解对话的 Agent：轻量问题改写和 Wiki 确认使用 Flash，证据判断与回答使用 Pro。系统 Prompt 告诉它目标、可以使用的工具和必须遵守的边界；Agent 根据用户这一轮说了什么、上一步做了什么、检索到了什么，自己选择下一个工具。

```text
用户提问
  → Agent 改写为一个更清楚的法规检索问题
  → 用户确认或直接纠正
  → 读取相关专家 Wiki，校准术语和业务场景
  → Agent 用一个完整问题调用混合检索
  → Agent 阅读 Chunk，判断证据是否足够
  → 必要时再检索一轮，并保留第一轮的相关证据
  → 同一个 Agent 基于最终证据撰写回答
  → 程序只检查引用是否真实、ID 是否对得上
  → 通过后展示法规名称、条文、官网链接和回答
```

## 二、问题改写与用户确认

用户首次提问后，Agent 不会立即检索。它先使用 `present_rewritten_question` 工具，把口语化提问改成一个主体、产品和待解决问题更清楚的完整问法。页面会把改写结果展示给用户，请用户确认。

后续回复由 Agent 按自然语言理解，没有“是就跳 A、不是就跳 B”的关键词路由：

- 用户确认：以已展示的问题检索。
- 用户直接给出正确问法：以新问法检索。
- 用户只说“不对”，却没说想问什么：Agent 使用 `ask_user` 继续追问。
- 用户明确指出上一回答哪里不对，并给出正确业务口径：Agent 整理为 Wiki 候选，必须再次得到用户确认才写入。

每个聊天在服务端保留独立 `sessionId`。点击“新对话”会新建一条独立历史对话，原对话、回答和依据仍保留在左侧列表中；不同对话不会共用 `sessionId`。

## 三、混合检索

Agent 使用唯一的法规检索工具 `hybrid_regulation_search`。每次调用只允许传入一个完整问题，不先拆成四个子问题，也不一次赛进多组同义检索词。

检索工具内部仍使用已经建好的两个通道：

- BM25 找文号、专业术语、比例、金额和明确措辞等字面命中。
- 本地 `bge-base-zh-v1.5` 向量找表达不同但意思相近的 Chunk。
- RRF 将两个 Chunk 排名等权融合，再去重和补充必要上下文。

整篇法规 BM25 不参与主排序，避免长文件中的通用词抬高无关 Chunk。融合后每份法规最多保留3个得分最高的 Chunk，避免单一长文件占满10条上下文；这项规则按 `documentId` 统一执行，不按题目或法规名称设置例外。

第一轮最多返回 10 个 Chunk。Agent 阅读后自己判断证据是否足够。证据不足时，它可以提出一个更准确的完整问题再检索一次。第二轮调用时，Agent 同时列出第一轮中仍相关的 evidence ID。程序保留这些 Chunk，再补入第二轮新结果；合并后的最终上下文仍最多为10个 Chunk，且同一法规在两轮合计中也不超过3个。

## 四、专家 Wiki 的作用边界

Wiki 保存用户在真实问答中确认的术语解释、实践经验和适用边界。新条目必须经过“模型识别候选→页面展示→用户明确确认→本地写入”，不会因用户一句“不对”就自动入库。

每次检索前，系统按当前问题找回少量相关 Wiki 条目。它们可以帮助 Agent 理解业务口径、保持检索问题准确，但不能覆盖法规原文，也不能单独支持允许、禁止、比例或期限等确定性法律结论。Wiki 与法规冲突时，以法规原文为准，并将 Wiki 标记为待复核。

## 五、证据足不足的处理

证据充分性不是单独的写死程序，也不是一个额外的审查模型。Agent 看完检索结果后，通过它选择“继续检索”还是“提交回答”来表达判断。

没有直接条文不代表系统必须失败。它可以回答：

1. 根据当前知识库检索到的法规，无法得出确定结论。
2. 现有法规明确规定到了哪一步。
3. 为什么这些条文还不能直接回答用户的问题。
4. 还需要查明什么业务事实或找到什么直接规定。
5. 本次参考了哪些法规、哪些条文，以及它们的官网链接。

因此最终回答不再被强制分为“是 / 否 / 不能确认”三种枚举。

## 六、回答与引用校验

同一个 Agent 调用 `submit_regulatory_answer` 提交：

- 直接结论；
- 可供用户理解和核验的理由摘要；
- 它选择的 evidence ID；
- 对应 Chunk 中的连续逐字原文；
- 每条原文与问题的关系；
- 回答确实使用的专家 Wiki 条目；未使用时为空。

Agent 不填法规标题、文号、发文机关、条号和官网 URL。这些字段全部由程序根据 evidence ID 从检索 Chunk 的真实元数据回填。

`CitationValidatorService` 现在只检查客观可确定的事：

- evidence ID 是否属于本次最终上下文；
- `quoteExact` 是否真的是对应 Chunk 中的连续逐字原文；
- 展示的法规名称、发文机关、文号、条号、效力状态和链接是否来自该 Chunk 的元数据。

它不审查“法律结论是否正确”，也不判断“这条原文是否足以支持结论”。如果真实性校验失败，程序把具体错误返回给 Agent，只允许它修订一次；第二次仍失败时不再循环，也不输出引用仍不真实的回答。

## 七、多对话运行状态

前端按 `conversationId` 隔离请求状态。一个对话运行时，用户可以切到其他对话继续输入并发起请求；切换页面不会取消后台回答。当前对话的输入框也保持可编辑并单独保存草稿，但同一对话一次只执行一个请求，防止服务端 session 状态相互覆盖。

## 八、上限和日志

必要的程序边界不是业务路由，而是防止无限调用和上下文膨胀：

- 一个回答最多两轮检索；
- 最终提供给回答 Agent 的 Chunk 最多 10 个；
- 引用校验失败后最多修订一次；
- 一次前端请求内最多执行 8 个 Agent 动作，防止异常循环；
- 这些上限对所有问题一致，没有三道示例题的专项规则或专项补丁。

每次对话的运行日志写入 `data/index/eval/logs/`，包括：

- 用户消息和 Agent 选择的工具；
- 每轮检索问题、目的、发现的 Chunk 和 BM25 / 向量 / RRF 排名；
- 第二轮保留了哪些第一轮证据；
- 引用校验失败的具体原因和修订次数；
- 最终结论、采用的 evidence ID、模型调用次数和耗时。

日志目录被 Git 忽略，不会将密钥或长上下文提交到仓库。

真实链路脚本的自动“通过”只验证流程完成与引用真实性，不替代法律结论的人工盲评。三道回归题和通用问题集均只存在于评测目录，不会进入 Prompt、检索加权或运行时规则。
