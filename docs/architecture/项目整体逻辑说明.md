# 中国场外衍生品法规知识库：项目整体逻辑说明

## 一、这个项目要解决什么问题

这个项目的目标，不是让大模型凭记忆回答法规问题，而是先从本地法规库中找到真正相关的条文，再让模型基于这些原文整理回答，并把法规名称、具体条文和官网链接一并展示出来。

整条链路可以概括为：

```text
法规原件
→ 结构化正文
→ 按法规结构切成 Chunk
→ 建立 BM25 关键词索引和本地向量索引
→ 用户提问
→ Agent 使用 Flash 轻量改写问题，并请用户确认
→ Agent 用一个完整问题调用混合检索工具
→ Agent 阅读检索到的 Chunk，必要时再检索一轮
→ Agent 基于合并后的证据撰写回答
→ 程序校验引用 ID 和逐字原文是否真实
→ 回填法规元数据和官网链接
→ 返回“结论 + 理由 + 法规原文 + 官网链接 + 证据边界”
```

系统的核心是三件事：

1. 法规原件转换成结构化正文时不丢内容、不改原意；
2. 用户提问后能找回真正关键的 Chunk；
3. 模型的引用必须来自本次检索结果，不能虚构法规或伪造原文。

## 二、法规库里有什么

当前正式知识库包括：

- 108 份法规原件；
- 108 份结构化正文；
- 1,228 个正式 Chunk；
- 108 份法规均配置官网链接；
- Chunk ID 唯一，且每个 Chunk 都能追溯到所属法规和结构化原文块。

静态的[法规与 Chunk 查看器](../法规与Chunk查看器.html)可以查看：

- 法规名称、文号、发文主体、效力状态和文件格式；
- 每份法规对应的 Chunk 数量；
- 每个 Chunk 的正文、条款范围和字数；
- 法规的官网链接；
- 按关键词、发文主体、效力状态和文件格式筛选。

法规的发布主体主要包括：

- 中国证监会；
- 中国证券业协会；
- 中国证券投资基金业协会；
- 中国期货业协会；
- 上海证券交易所和深圳证券交易所；
- 中国人民银行；
- 国家金融监督管理总局及其历史承接机构；
- 中国银行间市场交易商协会和中国外汇交易中心；
- 国务院和国家法律。

查看器左侧为了浏览方便，联合发文先按原字段的第一个主体归类，历史银保监机构归入国家金融监督管理总局。这只改变导航分类，法规详情仍完整保留历史发文主体。

## 三、法规原件如何变成结构化正文

原始法规包括 PDF、DOC 和 DOCX。不同格式先由对应解析器读取，再统一整理成结构化 JSON。

这一层主要处理：

1. 提取正文、表格、Word 文本框和 smartTag 中的有效内容；
2. 识别标题、文号、发文机关、发布日期、施行日期和效力状态；
3. 识别章、节、条、款、项、附件和表格；
4. 删除真正的目录、重复页眉页脚和孤立页码；
5. 合并 PDF 跨行、跨页导致的句内换行，但保留条款和列表边界；
6. 检查扫描页、乱码、私有区字符和公式丢失；
7. 给每个结构化原文块分配 `source_block_id`。

原始法规始终只读。清洗、排版恢复和层级整理都发生在结构化层，不直接修改原件。

## 四、Chunk 如何切分

一份法规可能有几十页。如果每次都把整份文件交给检索和模型，关键条文很容易被淹没，也不利于精确引用。因此项目先识别法规层级，再按语义单元切分，不是每隔固定字数粗暴切断。

边界优先级是：

```text
新文件或新分部
→ 章、节
→ 完整条款
→ 完整款、项或列表
→ 附件和表格
→ 完整句子和明显主题边界
```

当前长度约束大致为：

- 目标长度约 600 个正文字符；
- 一般上限约 1,200 个正文字符；
- 低于 200 字且不是独立完整条款的普通短块，优先和相邻内容合并。

长度只是约束，完整语义优先。系统尤其保护：

- “包括以下内容”“应符合下列条件”等引导语不与列表分离；
- 一个完整句子不从中间切开；
- 表格优先按完整行处理，不从单元格中间截断；
- “依照前款”“除前项外”等明显依赖前文的内容保留必要上下文；
- overlap 只在必要时使用，不做机械滑窗重复。

每个 Chunk 除了正文，还保留：

- `chunk_id`、`document_id` 和文档内顺序；
- 法规名称、文号和发文机关；
- 发布日期、施行日期和效力状态；
- 章、节、条款范围和附件名称；
- 官网链接；
- 对应的 `source_block_id`；
- overlap 来源和是否存在不可避免的超长语义单元。

## 五、BM25 关键词检索

BM25 擅长找：

- 法规名称和文号；
- 专业术语；
- 数字、比例、金额和期限；
- “不得”“应当”“除外”等明确措辞。

中文文本使用 NFKC 规范化、中文双字切分和场外衍生品领域词识别。标题、章节、条号和正文进入同一个索引，不因发文机关级别人为加权。

BM25 的优点是对原文用词和数字非常敏感；局限是用户的口语表达可能和法规正式语言不同。

## 六、向量检索

向量检索用于找“说法不同，意思相近”的 Chunk。例如，用户说“券商收益凭证做雪球”，法规可能写的是“证券公司发行浮动收益凭证”“挂钩标的”“内嵌衍生品”或“敲入敲出”。

当前本地向量模型为 `Xenova/bge-base-zh-v1.5`，768 维，q8 量化。它只负责把问题和 Chunk 转成可比较的向量，不生成法律结论，也不需要把全部法规发送给外部大模型。

Chunk 内容改变后，向量必须重新生成，旧向量不能继续配新 Chunk。

## 七、混合检索

每次 `hybrid_regulation_search` 工具调用会同时执行：

1. BM25 取前 30 条；
2. 向量检索取前 30 条；
3. 用等权 RRF 融合两个排名；
4. 同一法规最多保留3个得分最高的 Chunk，避免单一长法规占满10条回答上下文；
5. 去重，并补齐必要的前置条款和 overlap 上下文；
6. 返回最多 10 个 Chunk 给 Agent。

RRF 可以理解为“排名投票”。同一个 Chunk 在关键词和语义检索中都靠前，融合后通常也会靠前。它不直接相加两套口径不同的原始分数。

## 八、对话式法规 Agent

### 1. 为什么只用一个主 Agent

旧问答链路中，“规划、证据判断、回答、独立审查”是多个固定步骤。现在这套状态机已删除。

当前是一个连续对话的 DeepSeek Agent。问题改写和 Wiki 确认等轻任务使用 Flash，证据判断与回答使用 Pro。System Prompt 告诉它应达到什么目标、可以使用哪些工具以及哪些边界不能突破。Agent 根据完整对话和工具结果自己决定下一步，不由“用户说了哪个关键词”的写死路由决定。

### 2. 先改写问题，再让用户确认

新对话的第一轮，Agent 只做问题改写，不检索。改写的目标是保留用户原意，同时把主体、产品、行为和待解决的法规问题说清楚。

用户的回复由同一个 Agent 按自然语言理解：

- 如果用户确认，Agent 使用已改写的完整问题检索；
- 如果用户直接给出了修正后的问题，Agent 使用新问题；
- 如果用户只说“不对”却没有说正确问法，Agent 继续追问。
- 如果用户明确纠正上一回答并给出新的业务口径，Agent 先展示 Wiki 候选；只有用户再次确认后才写入。

### 3. 检索一个完整问题

确认后，Agent 每轮只把一个完整问题传给混合检索，不会先拆成四个子问题，也不一次输入多个专项检索词。

第一轮返回最多 10 个 Chunk。Agent 阅读它们后，自己判断：

- 哪些 Chunk 与问题相关；
- 现有证据是否足够回答；
- 如果不足，到底缺少什么直接规定；
- 是否能用一个更准确的完整问题再检索一轮。

最多只能检索两轮。如果进行第二轮，Agent 需要同时列出第一轮中仍相关的 evidence ID。系统保留这些原文，再加入第二轮新结果。合并后的最终上下文仍最多只有 10 个 Chunk，避免第二轮覆盖第一轮，也避免上下文无限膨胀。

### 4. 证据不足也是正常回答

如果当前法规只规定了一般条件，却没有直接写明用户问的具体产品“可以”或“不可以”，Agent 不应该为了交付一个简单结论而强行推导。

这时回答可以是：

> 根据当前知识库检索到的法规，无法得出确定结论。现有条文明确了……，但并未直接规定……。

然后继续列出相关法规、逐字条文、官网链接和仍缺少的信息。最终回答不再被强制分为“是 / 否 / 不能确认”三个枚举选项。

### 5. 回答如何组织

Agent 提交的回答包含：

1. 直接结论；
2. 通俗、可核验的理由摘要；
3. 它选择的 evidence ID；
4. 从对应 Chunk 中连续逐字复制的原文；
5. 每条原文与用户问题的关系和边界；
6. 回答实际使用的专家 Wiki 条目；未使用时为空。

模型不自行生成法规标题、文号、发文机关、条号、效力状态和官网 URL。这些都由系统根据 evidence ID 从真实 Chunk 元数据回填，因此不会出现“模型说它引用 A 法规，实际原文来自 B 法规”的情况。

## 九、程序只检查引用真实性

最终回答不再调用独立审查模型。程序化校验也不判断法律结论是否正确，不判断该条文是否足以支持模型的结论，也不自动把回答改成“是”“否”或“不能确认”。

`CitationValidatorService` 只检查三件客观可确定的事：

1. evidence ID 是否真的属于本次提供给 Agent 的最终 Chunk 上下文；
2. `quoteExact` 是否是对应 Chunk 中的连续逐字原文；
3. 法规名称、文号、发文机关、条号、效力状态和官网链接是否由该 evidence ID 的真实元数据回填。

如果第一次校验失败，程序把具体错误返回给 Agent，让它只修正错误的 ID 或引文。最多只修订一次。如果第二次仍失败，系统不再循环，也不输出引用真实性仍未通过的回答。

## 十、三个真实问题如何理解

下面的三道问题是手动测试示例，不是程序里的专项路由，也没有预先标注该召回哪些 Chunk。每道题都要先经用户确认，再走同一套通用 Agent 和混合检索逻辑。

### 1. 上市公司可以做挂钩自己本身标的的场外衍生品吗？

这个口语问题需要弄清“自己本身标的”是不是指上市公司自身发行的股票，以及场外衍生品的交易对手是证券公司、期货风险管理公司还是其他持牌机构。检索到的现行条文和已公布但尚未施行的统一办法，还需要区分生效时间。

回答不应把某一类交易对手的禁止自动扩大成“所有交易对手一律禁止”，但可以分别展示已检索到的相关制度及其适用范围。

### 2. 券商收益凭证可以做雪球吗？

如果只检索到浮动收益凭证的一般发行、挂钩标的、业务资格、损失结构和信息披露规则，却没有直接条文写明“雪球收益凭证可以发行”或“不得发行”，就不应只凭一般条件无条件回答“可以”。

更合理的回答是：说明当前检索结果无法支持一个普遍、确定结论，再列出现有法规已经规定的一般边界、具体条文和官网链接，由业务人员结合产品结构和最新监管口径继续判断。

### 3. 私募产品投资雪球比例的范围有明确规定吗？

需要先区分口语中的“私募产品”到底是不是私募证券投资基金。《私募证券投资基金运作指引》对带敲入和敲出结构的场外期权或收益凭证，包括雪球结构，有明确的 25% 原则上限和严格例外条件。

这一规则直接适用于私募证券投资基金，不能未经核实就自动扩大到所有私募股权基金、资产管理计划或其他私募产品。

## 十一、日志和手动测试

每个聊天会产生独立的 `sessionId` 和 `runId`。服务端把运行过程写入 `data/index/eval/logs/`，包括：

- 用户每轮输入；
- Agent 每次选择的工具；
- 检索问题、检索目的、命中 Chunk 和 BM25 / 向量 / RRF 排名；
- 第二轮保留的第一轮证据；
- 引用校验失败的具体原因；
- 最终回答、采用的 evidence ID、模型调用次数和每次耗时。

日志目录被 Git 忽略，不会提交 API Key 或长法规上下文。

三道真实问题不再由自动评测脚本强制判断“通过 / 失败”。启动本地 API 和聊天页面后，按真实用户对话手动测试，并结合页面回答和运行日志判断问题出在改写、检索、证据保留还是引用生成。

## 十二、如何理解项目的可靠性

这个项目不是因为“大模型很聪明”才可靠，而是通过分层降低错误：

```text
原件负责提供真实内容
结构化层负责保持正文、表格和层级
Chunk 层负责提供合适的检索单位
BM25 负责找准专业词、原文措辞和数字
向量模型负责理解近义和口语表达
RRF 负责融合两类排名
主 Agent 负责理解用户、选择工具、判断证据和组织回答
引用校验负责阻止伪造 evidence ID 和伪造逐字原文
人工判断最终处理法律结论、业务事实和监管口径的边界
```

必要的程序上限只用于防止异常循环和上下文膨胀：最多两轮检索、最多 10 个最终 Chunk、最多一次引用修订。这些边界对所有问题一致，没有为某三道问题定制的分支或补丁。

最后，本系统提供法规检索、原文引用和证据整理，不替代律师法律意见或金融机构内部合规审批。
